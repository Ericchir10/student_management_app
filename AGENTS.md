# Agent Instructions for Student Management App

## Project Overview
Flutter application for student management with SQLite backend. Single-database architecture with 8 screens, no external state management (direct DB calls from UI).

**Key Stack**: Flutter 3.5+, sqflite 2.3.0, Material Design  
**Target**: Android, iOS, Web  

---

## File Organization & Exclusions

### ✅ Active Source Directories (Modify These)
- **`lib/screens/`** — UI pages using StatelessWidget/StatefulWidget
- **`lib/models/`** — Data models (`Student`, `UserModel`)
- **`lib/database/database_helper.dart`** — Singleton managing SQLite operations
- **`lib/main.dart`** — App entry point with named routing configuration
- **`test/`** — Widget tests

### ❌ Do Not Modify (Generated/Build)
- **`build/`** — Build artifacts; auto-generated by Flutter
- **`android/app/{debug|profile|release}`** — Platform-specific builds
- **`.dart_tool/`** — Dart compiler cache; regenerated on `flutter pub get`
- **`.flutter-plugins-dependencies`** — Plugin manifest; auto-generated
- **`ios/Flutter/ephemeral/`** — Ephemeral generated files
- **`linux/`, `macos/`, `windows/`** — Platform scaffolding; modify only via Flutter tooling
- **`*.iml`, `.idea/`** — IDE metadata

### ⚠️ Legacy/Deprecated (Avoid Using)
- **`lib/legacy_code/student_service.dart`** — Deprecated; duplicate DB logic
  - **Action**: Use `DatabaseHelper` directly from screens instead
- **Duplicate `UserModel` class** — Functionally identical to `Student`
  - **Action**: Consolidate to single canonical model

### 📋 Special Files
- **`.gitignore`** — Root and platform-specific ignore patterns; consult before excluding new files
- **`pubspec.yaml`** — Dependency & build configuration; read for context, modify carefully
- **`analysis_options.yaml`** — Lint rules; reflects code quality expectations

## Key Conventions

### Navigation & Routing
All navigation uses **named routes** defined in `main.dart`:
```dart
Navigator.pushNamed(context, '/dashboard');  // ✅ Correct
Navigator.push(context, MaterialPageRoute(...));  // ❌ Avoid
```
**Route keys**: `/login`, `/dashboard`, `/students`, `/add-student`, `/attendance`, `/report`

### Database Operations
- **Single instance**: `DatabaseHelper` initialized in `main()` via `WidgetsFlutterBinding.ensureInitialized()`
- **All async**: DB operations return `Future<T>`
- **Serialization**: Use `.toMap()` for insert/update, `.fromMap()` for queries
- **Tables**: `students`, `attendance` (see `DatabaseHelper` for schema)

### State Management
**No state management framework used** (no Provider, Bloc, Riverpod).
- Screens call `DatabaseHelper` methods directly
- UI refreshes via `StatefulWidget.setState()` or `FutureBuilder`

### File Patterns
- **Screens**: `*_screen.dart` with class name `XyzScreen` extending Widget
- **Models**: Plural nouns (`student.dart`, `user_model.dart`)
- **Services**: `*_service.dart` (empty by design; use DatabaseHelper instead)

## Workflow Guidance

### Before Modifying Files
1. **Check file source**: Is it in `lib/` (modify) or `build/`/`.dart_tool/` (regenerate)?
2. **For platform code**: Modify `android/app/src/`, `ios/Runner/` via Xcode/Android Studio
3. **For dependencies**: Edit `pubspec.yaml`, then run `flutter pub get`

### Regenerating Excluded Files
```bash
flutter pub get          # Restore .pub-cache and .dart_tool
flutter clean            # Remove build/ and regenerate
flutter pub global activate<package>  # Regenerate tools
```

### Code Quality
Run before committing:
```bash
flutter analyze          # Check lint rules (analysis_options.yaml)
flutter test             # Run widget tests (test/*.dart)
flutter format lib/      # Auto-format Dart code
```

---

## Common Pitfalls

1. **Modifying generated files** (e.g., in `build/`, `.dart_tool/`) — Changes lost on rebuild; regenerate instead
2. **Using `StudentService` from legacy_code** — Contains duplicate DB logic; use `DatabaseHelper` directly
3. **Direct navigation** (MaterialPageRoute) instead of named routes — Breaks app routing consistency
4. **Sync DB calls** — Flutter requires `Future<T>`; never use `.blockingXyz()`
